## 3. 各エンドポイント処理詳細

### 3-1. GET /api/foods — お品一覧取得

#### クエリパラメータ

| パラメータ | 型 | デフォルト | 説明 |
|---|---|---|---|
| `name` | string | — | お品名（部分一致） |
| `sortBy` | string | `name` | `name` / `price` / `created_at` |
| `sortOrder` | string | `ASC` | `ASC` / `DESC` |
| `limit` | integer | 20 | 最大 100 |
| `offset` | integer | 0 | — |

#### 処理フロー

1. JWT を検証する
2. クエリパラメータをバリデーションする
3. `item_type = 3` で絞り込み、`items` テーブルを検索する
4. `{ success, data, count, timestamp }` を返す（構成仕込み品リストは含めない）

#### DB操作

```sql
SELECT id, name, item_type, store, price, quantity, unit, unit_price,
       genre, description, created_at, updated_at
FROM items
WHERE item_type = 3
  AND (:name IS NULL OR name LIKE CONCAT('%', :name, '%'))
ORDER BY :sortBy :sortOrder
LIMIT :limit OFFSET :offset;
```

### 3-2. POST /api/foods — お品登録（確定）

#### バリデーションルール

| フィールド | 必須 | ルール |
|---|:---:|---|
| `name` | ✓ | 1〜255文字 |
| `quantity` | ✓ | 数値、`> 0`（提供数量、通常 1.0） |
| `unit` | ✓ | 1〜20文字（例: 人前） |
| `genre` | — | 50文字以内 |
| `description` | — | 任意文字列 |
| `items` | ✓ | 配列、1件以上 |
| `items[].prep_id` | ✓ | 正の整数、`items` テーブルに存在する仕込み品（`item_type = 2`） |
| `items[].amount` | ✓ | 数値、`> 0` |
| `items[].description` | — | 任意文字列 |

#### 処理フロー

1. JWT を検証する
2. リクエストボディをバリデーションする
3. `items[]` の全 `prep_id` が `items (item_type = 2)` に存在するか確認する
4. 各仕込み品の `unit_price` を取得し、`cost = unit_price × amount` を計算する
5. `price（食材原価）= 全 cost の合計` を算出する
6. `unit_price = price ÷ quantity` を計算する（お品自体の単価）
7. トランザクション内で以下を実行する：
   - `items` に `item_type = 3`、`store = '自家製'` でレコードを `INSERT` する
   - `item_relations` に各仕込み品 `(parent_id=食id, child_id=prep_id, amount, cost, description)` を `INSERT` する
8. 作成されたお品レコードを返す

#### DB操作

```sql
BEGIN;

INSERT INTO items (name, item_type, store, price, quantity, unit, unit_price, genre, description, created_at, updated_at)
VALUES (:name, 3, '自家製', :total_cost, :quantity, :unit, :unit_price, :genre, :description, NOW(), NOW());

INSERT INTO item_relations (parent_id, child_id, amount, cost, description, created_at)
VALUES (:food_id, :prep_id, :amount, :cost, :description, NOW());

COMMIT;
```

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| バリデーション失敗 | 400 | `VALIDATION_ERROR` |
| 存在しない `prep_id` | 400 | `VALIDATION_ERROR` |

### 3-3. GET /api/foods/stats/profit — 利益率順お品取得

#### クエリパラメータ

| パラメータ | デフォルト | 最大 |
|---|---|---|
| `limit` | 10 | 50 |

#### 処理フロー

1. JWT を検証する
2. `item_type = 3` の全お品を取得する
3. 現フェーズでは `items.price` が食材原価のみのため、利益率計算には別途販売価格が必要
4. 販売価格が未設定の場合は `price`（食材原価）の低い順を「効率的」として返す

> **注意**: 将来的に `selling_price` カラムが追加された場合は `(selling_price - price) / selling_price × 100` で利益率を算出する。

#### DB操作

```sql
SELECT id, name, price, quantity, unit, unit_price, genre, description
FROM items
WHERE item_type = 3
ORDER BY unit_price ASC   -- 現フェーズ: 原価の低い順
LIMIT :limit;
```

### 3-4. GET /api/foods/{id} — お品詳細取得

#### 処理フロー

1. JWT を検証する
2. `SELECT * FROM items WHERE id = :id AND item_type = 3` でお品を取得する（404 チェック）
3. `item_relations` を JOIN して構成仕込み品リストを取得する
4. お品情報 + `preps[]` 配列を返す
5. 販売価格シミュレーターはこのエンドポイントの `price`（食材原価）を参照する

#### DB操作

```sql
-- お品基本情報
SELECT id, name, item_type, store, price, quantity, unit, unit_price,
       genre, description, created_at, updated_at
FROM items
WHERE id = :id AND item_type = 3;

-- 構成仕込み品一覧
SELECT
  ir.child_id AS prep_id,
  i.name AS prep_name,
  i.genre AS prep_genre,
  i.unit_price AS prep_unit_price,
  ir.amount,
  ir.cost,
  ir.description
FROM item_relations ir
JOIN items i ON ir.child_id = i.id
WHERE ir.parent_id = :id
ORDER BY ir.id ASC;
```

### 3-5. PUT /api/foods/{id} — お品更新

#### バリデーションルール（変更フィールドのみ送信可）

| フィールド | 必須 | ルール |
|---|:---:|---|
| `name` | — | 255文字以内 |
| `quantity` | — | 数値、`> 0` |
| `unit` | — | 20文字以内 |
| `genre` | — | 50文字以内 |
| `description` | — | 任意文字列 |
| `items` | — | 配列、1件以上（送信時は全件置き換え） |

#### 処理フロー

1. JWT を検証する
2. `SELECT id FROM items WHERE id = :id AND item_type = 3` で存在確認する（404 チェック）
3. リクエストボディをバリデーションする
4. `items[]` が送信された場合：
   - 全 `prep_id` の存在確認を行う
   - 各仕込み品の `cost` を再計算する
   - `price`（食材原価）を再計算する
   - `unit_price` を再計算する
   - 既存 `item_relations (parent_id = :id)` を全件 `DELETE` し、新しい構成で `INSERT` する
5. トランザクション内で `items` の更新と `item_relations` の入れ替えを行う
6. 更新後のお品レコード（`preps[]` 含む）を返す

#### DB操作

```sql
BEGIN;

-- 旧構成仕込み品を削除
DELETE FROM item_relations WHERE parent_id = :id;

-- 新構成仕込み品を登録
INSERT INTO item_relations (parent_id, child_id, amount, cost, description, created_at)
VALUES (:id, :prep_id, :amount, :cost, :description, NOW());

-- お品本体を更新
UPDATE items
SET name = :name, quantity = :quantity, unit = :unit,
    price = :new_total_cost, unit_price = :new_unit_price,
    genre = :genre, description = :description, updated_at = NOW()
WHERE id = :id AND item_type = 3;

COMMIT;
```

### 3-6. DELETE /api/foods/{id} — お品削除

#### 処理フロー

1. JWT を検証する
2. `SELECT id FROM items WHERE id = :id AND item_type = 3` で存在確認する（404 チェック）
3. トランザクション内で以下を実行する：
   - `item_relations WHERE parent_id = :id` を `DELETE`（CASCADE 未設定の場合）
   - `items WHERE id = :id AND item_type = 3` を `DELETE`
4. `{ success: true, message: "お品を削除しました" }` を返す

> お品は他のお品から参照されることはないため、削除前の使用中チェックは不要。

#### DB操作

```sql
DELETE FROM item_relations WHERE parent_id = :id;
DELETE FROM items WHERE id = :id AND item_type = 3;
```

---

## 4. 設計注意事項

- お品登録・更新は必ずトランザクションで行う（`items` と `item_relations` の整合性保証）
- コスト計算（`prep.unit_price × amount`）はサーバー側のみで行い、クライアント送信値を使わない
- `items[]` を送信した更新は全件置き換え方式（差分更新は行わない）
- `store` カラムは常に `'自家製'` を固定でセットする
- `item_type = 3` の絞り込みを全クエリに必ず付ける
