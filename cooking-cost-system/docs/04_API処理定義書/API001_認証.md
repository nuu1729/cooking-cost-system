## 3. 各エンドポイント処理詳細

### 3-1. POST /api/auth/register — アカウント登録

#### バリデーションルール

| フィールド | 必須 | ルール |
|---|:---:|---|
| `username` | ✓ | 3〜50文字、英数字・アンダースコア・ハイフンのみ |
| `email` | ✓ | RFC 5322準拠メールアドレス形式 |
| `password` | ✓ | 8〜100文字 |

#### 処理フロー

1. リクエストボディを受け取り、バリデーションを実行する
2. `users` テーブルで `username` の重複チェック（`SELECT id FROM users WHERE username = :username`）
3. `users` テーブルで `email` の重複チェック（`SELECT id FROM users WHERE email = :email`）
4. `bcrypt`（コストファクター ≥ 12）でパスワードをハッシュ化する
5. `users` テーブルに新規レコードを `INSERT` する
6. JWT を生成し（有効期限 24h）、`{ user, token, expiresAt }` を返す

#### DB操作

```sql
-- 重複チェック
SELECT id FROM users WHERE username = :username;
SELECT id FROM users WHERE email = :email;

-- 登録
INSERT INTO users (username, email, password_hash, created_at, updated_at)
VALUES (:username, :email, :password_hash, NOW(), NOW());
```

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| バリデーション失敗 | 400 | `VALIDATION_ERROR` |
| username 重複 | 409 | `CONFLICT` |
| email 重複 | 409 | `CONFLICT` |

### 3-2. POST /api/auth/login — ログイン

#### バリデーションルール

| フィールド | 必須 | ルール |
|---|:---:|---|
| `email` | ✓ | 文字列（空文字不可） |
| `password` | ✓ | 文字列（空文字不可） |

#### 処理フロー

1. リクエストボディを受け取り、バリデーションを実行する
2. `SELECT * FROM users WHERE email = :email` でユーザーを取得する
3. ユーザーが存在しない場合は 401 を返す（ユーザー特定を防ぐため email/password 両方を「認証失敗」として返す）
4. `bcrypt.compare(password, password_hash)` でパスワードを照合する
5. 照合失敗時は 401 を返す
6. JWT を生成し（有効期限 24h）、`{ user, token, expiresAt }` を返す

#### DB操作

```sql
SELECT id, username, email, password_hash, is_active
FROM users
WHERE email = :email;
```

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| バリデーション失敗 | 400 | `VALIDATION_ERROR` |
| メール未登録 または パスワード不一致 | 401 | `UNAUTHORIZED` |
| アカウント無効（is_active = false） | 401 | `UNAUTHORIZED` |

---

### 3-3. POST /api/auth/logout — ログアウト

#### 処理フロー

1. `Authorization: Bearer <token>` ヘッダーから JWT を検証する
2. JWT はステートレスのため、サーバー側での無効化処理は行わない
3. クライアントは `localStorage` から `authToken` を削除することでログアウト完了
4. `{ success: true, message: "ログアウトしました" }` を返す

> **注意**: 現フェーズでは JWT ブラックリストを管理しない。将来的にトークン即時無効化が必要な場合は Redis 等でブラックリスト管理を追加する。

---

### 3-4. GET /api/auth/me — 認証ユーザー情報取得

#### 処理フロー

1. `Authorization: Bearer <token>` ヘッダーから JWT を検証する
2. JWT の `sub`（user_id）で `SELECT id, username, email, created_at, updated_at FROM users WHERE id = :user_id` を実行する
3. ユーザー情報（`password_hash` を除く）を返す

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| トークン未指定 / 不正 / 期限切れ | 401 | `UNAUTHORIZED` |
| ユーザーが削除済み | 401 | `UNAUTHORIZED` |

---

### 3-5. GET /api/auth/status — トークン有効性確認

#### 処理フロー

1. JWT を検証する
2. 有効な場合は `{ valid: true, user: { id, username } }` を返す
3. 無効または期限切れの場合は `{ valid: false }` を 200 で返す（エラーにしない）

> フロントエンドの初回マウント時にトークン有効性を確認するために使用する。

---

### 3-6. POST /api/auth/refresh — トークンリフレッシュ

#### 処理フロー

1. 現在のJWT を検証する（期限切れでも署名が正しければ許可する実装を推奨）
2. JWT の `sub` でユーザー存在確認を行う
3. 新しい JWT を生成し（有効期限 24h）、返す

---

### 3-7. PUT /api/auth/profile — プロフィール更新

#### バリデーションルール

| フィールド | 必須 | ルール |
|---|:---:|---|
| `username` | — | 3〜50文字、英数字・アンダースコア・ハイフン |
| `email` | — | RFC 5322準拠 |

#### 処理フロー

1. JWT を検証し、`user_id` を取得する
2. リクエストボディのバリデーションを実行する
3. `username` が送信された場合、他ユーザーとの重複チェックを行う（`WHERE username = :username AND id != :user_id`）
4. `email` が送信された場合、他ユーザーとの重複チェックを行う（`WHERE email = :email AND id != :user_id`）
5. `UPDATE users SET ... WHERE id = :user_id` で更新する
6. 更新後のユーザー情報（`password_hash` を除く）を返す

#### DB操作

```sql
-- 重複チェック（username）
SELECT id FROM users WHERE username = :username AND id != :user_id;

-- 更新
UPDATE users
SET username = :username, email = :email, updated_at = NOW()
WHERE id = :user_id;
```

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| トークン不正 | 401 | `UNAUTHORIZED` |
| バリデーション失敗 | 400 | `VALIDATION_ERROR` |
| username / email 重複 | 409 | `CONFLICT` |

---

### 3-8. PUT /api/auth/password — パスワード変更

#### バリデーションルール

| フィールド | 必須 | ルール |
|---|:---:|---|
| `current_password` | ✓ | 文字列（空文字不可） |
| `new_password` | ✓ | 8〜100文字 |

#### 処理フロー

1. JWT を検証し、`user_id` を取得する
2. `SELECT password_hash FROM users WHERE id = :user_id` を実行する
3. `bcrypt.compare(current_password, password_hash)` で現在のパスワードを照合する
4. 照合失敗時は 401 を返す
5. `new_password` を bcrypt（コストファクター ≥ 12）でハッシュ化する
6. `UPDATE users SET password_hash = :new_hash, updated_at = NOW() WHERE id = :user_id` で更新する
7. `{ success: true, message: "パスワードを変更しました" }` を返す

#### エラーパターン

| 条件 | ステータス | error コード |
|---|---|---|
| トークン不正 | 401 | `UNAUTHORIZED` |
| バリデーション失敗 | 400 | `VALIDATION_ERROR` |
| 現在のパスワード不一致 | 401 | `UNAUTHORIZED` |

## 4. 共通処理

### JWT 検証ミドルウェア

全認証必須エンドポイントに適用する共通ミドルウェア。

1. `Authorization` ヘッダーから `Bearer <token>` を取り出す
2. `JWT_SECRET` で署名を検証する
3. 期限切れ（`exp`）を確認する
4. `sub`（user_id）を `request.user_id` にセットする
5. いずれかが失敗した場合は 401 を返す

### セキュリティ注意事項

- `password_hash` は絶対にレスポンスに含めない
- ログイン失敗時はユーザー特定を防ぐため「メールまたはパスワードが正しくありません」と返す
- bcrypt コストファクターは ≥ 12 とする
