## 2. テーブル一覧

食材の購入先（仕入れ先）を正規化して管理するマスタテーブル。  
従来 `items.store`（自由テキスト）で管理していた購入先を独立テーブルに分離し、  
`items.store_id` FK で参照する設計に変更。

| テーブル名 | 論理名 | 説明 |
|---|---|---|
| `stores` | 購入先マスタ | 購入先の名称をユーザーごとに管理する |

---

## 3. テーブル定義

### 3-1. `stores` テーブル（購入先マスタ）

#### カラム定義

| カラム名 | 論理名 | データ型 | NOT NULL | デフォルト値 | 備考 |
|---|---|---|:---:|---|---|
| `id` | 購入先ID | `INT` | ✓ | AUTO_INCREMENT | 主キー |
| `user_id` | ユーザーID | `INT` | ✓ | — | `users.id` への外部キー。CASCADE DELETE |
| `name` | 購入先名 | `VARCHAR(100)` | ✓ | — | 例：コスモス、業務スーパー、自家製 |
| `created_at` | 作成日時 | `DATETIME` | ✓ | `CURRENT_TIMESTAMP` | レコード作成日時 |
| `updated_at` | 更新日時 | `DATETIME` | ✓ | `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | レコード更新日時 |

#### 制約・インデックス定義

| 制約名 | 種別 | 対象カラム | 説明 |
|---|---|---|---|
| `PRIMARY KEY` | 主キー | `id` | 主キー制約 |
| `fk_stores_user` | 外部キー | `user_id → users.id` | ユーザー削除時に購入先も CASCADE DELETE |
| `ix_stores_user_id` | インデックス | `user_id` | ユーザー別絞り込みの高速化 |

> **一意性**: 同一ユーザー内での購入先名の重複はアプリケーション層でチェックする（DB UNIQUE 制約は未設定）。

---

### 3-2. `items` テーブルへの追加カラム（store_id）

食材（`item_type = 1`）に対して `store_id` FK を追加。  
仕込み品・お品（`item_type = 2, 3`）は `store = '自家製'` のままとし、`store_id = NULL` を許容する。

| カラム名 | 論理名 | データ型 | NOT NULL | デフォルト値 | 備考 |
|---|---|---|:---:|---|---|
| `store_id` | 購入先ID | `INT` | — | `NULL` | `stores.id` への外部キー。ON DELETE SET NULL |

#### 追加 DDL

```sql
ALTER TABLE items
  ADD COLUMN store_id INT NULL,
  ADD CONSTRAINT fk_items_store_id
    FOREIGN KEY (store_id) REFERENCES stores(id) ON DELETE SET NULL,
  ADD INDEX ix_items_store_id (store_id);
```

#### データ移行 SQL（初回のみ実行）

```sql
UPDATE items i
JOIN stores s
  ON i.store COLLATE utf8mb4_unicode_ci = s.name COLLATE utf8mb4_unicode_ci
  AND i.user_id = s.user_id
SET i.store_id = s.id
WHERE i.item_type = 1;
```

## 4. DDL（CREATE TABLE文）

```sql
CREATE TABLE IF NOT EXISTS `stores` (
  `id`         INT          NOT NULL AUTO_INCREMENT  COMMENT '購入先ID',
  `user_id`    INT          NOT NULL                 COMMENT 'ユーザーID（users.id）',
  `name`       VARCHAR(100) NOT NULL                 COMMENT '購入先名',
  `created_at` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP                          COMMENT '作成日時',
  `updated_at` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新日時',

  PRIMARY KEY (`id`),
  INDEX `ix_stores_user_id` (`user_id`),
  CONSTRAINT `fk_stores_user`
    FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci
  COMMENT='購入先マスタ（ユーザーごとに管理）';
```

## 5. 補足事項

### 5-1. store テキスト列との二重管理について

`items.store`（VARCHAR）は後方互換性のため残存する。  
食材の登録・更新時にバックエンドが `stores.name` を取得して `items.store` へ同期する。  
購入先名変更（`PUT /api/stores/{id}`）時も `items.store` を一括 UPDATE する。

| カラム | 役割 |
|---|---|
| `items.store_id` | 正規化された外部キー参照（推奨） |
| `items.store` | 表示用テキストキャッシュ・後方互換 |

### 5-2. 削除制約

食材（`item_type = 1`）で `store_id` が参照されている購入先は削除不可とする。  
削除時にアプリケーション層でカウントチェックを行い、1件以上あれば 409 を返す。  
（`ON DELETE SET NULL` FK は保険として設定するが、アプリ層で先行チェックする）

### 5-3. 仕込み品・お品の扱い

`item_type = 2`（仕込み品）と `item_type = 3`（お品）の `store` は常に `'自家製'` とし、  
`store_id = NULL` を維持する。購入先マスタの管理対象外。
