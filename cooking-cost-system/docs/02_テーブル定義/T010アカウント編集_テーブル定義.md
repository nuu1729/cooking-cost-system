## 3. テーブル詳細（参照・更新対象カラム）

### 3-1. `users` テーブル — アカウント編集で操作するカラム

> テーブルの全体定義は `T000 アカウント作成_テーブル定義.md` を参照。  
> 以下はアカウント編集処理で特に関係するカラムを抜粋して記載する。

| カラム名 | 論理名 | データ型 | 操作 | 備考 |
|---|---|---|:---:|---|
| `id` | ユーザーID | `INT` | 参照のみ | JWT の `sub` クレームと照合して編集対象を特定する |
| `username` | アカウント名 | `VARCHAR(50)` | 更新可 | 任意入力。最大 50 文字 |
| `email` | メールアドレス | `VARCHAR(255)` | 更新可 | 変更後は一意性チェック（`uk_email`）が発動する |
| `password_hash` | パスワードハッシュ | `VARCHAR(255)` | 更新可 | パスワード変更時のみ更新。bcrypt で再ハッシュした値を格納 |
| `updated_at` | 更新日時 | `TIMESTAMP` | 自動更新 | `ON UPDATE CURRENT_TIMESTAMP` により自動設定 |

#### クエリの概要

```sql
-- ① 現在のプロフィールを取得（初期表示用）
SELECT id, username, email, role, is_active, last_login, created_at, updated_at
FROM   users
WHERE  id = :user_id;

-- ② プロフィール更新（PUT /api/auth/profile）
UPDATE users
SET    username   = :username,
       email      = :email
WHERE  id = :user_id;

-- ③ パスワード更新（PUT /api/auth/password）
-- ※ 事前に現在の password_hash と入力値を bcrypt.verify で照合すること
UPDATE users
SET    password_hash = :new_password_hash
WHERE  id = :user_id;
```

## 5. 補足事項

### 5-1. プロフィール更新フロー（`PUT /api/auth/profile`）

```
① JWT から user_id を抽出・検証
       │
       ├─ 無効なトークン → 401（未認証）
       │
② メールアドレスが変更された場合
       ├─ 新しい email で重複チェック（uk_email）
       │       └─ 既存ユーザーと重複 → 409（メールアドレス重複）
       │
③ UPDATE users SET username = ?, email = ? WHERE id = ?
       │
④ 更新されたデータを取得して 200 で返却
```

### 5-2. パスワード更新フロー（`PUT /api/auth/password`）

```
① JWT から user_id を抽出・検証
       │
② users テーブルから password_hash を取得
       │
③ bcrypt.verify(currentPassword, password_hash)
       └─ 不一致 → 401（現在のパスワードが正しくありません）
       │
④ newPassword の複雑性ルール検証
       └─ 不合格 → 400（バリデーションエラー）
       │
⑤ bcrypt.hash(newPassword, cost=12) で新しいハッシュ生成
       │
⑥ UPDATE users SET password_hash = ? WHERE id = ?
       │
⑦ 200 で成功レスポンスを返却
```

### 5-3. アカウントアイコンの保存

| フェーズ | 保存先 | カラム |
|---|---|---|
| 現フェーズ | クライアント localStorage（`accountStore`） | DB カラムなし |
| 将来 | クラウドストレージ（S3 等） + `users.avatar_url` | `avatar_url VARCHAR(512) NULL` を追加予定 |

> 将来移行時は `T000 アカウント作成_テーブル定義.md` も合わせて更新すること。

### 5-4. セキュリティ

| 観点 | 方針 |
|---|---|
| 権限チェック | JWT の `sub`（user_id）と `WHERE id = :user_id` を必ず照合し、他ユーザーの情報を書き換えられないようにする |
| パスワードの平文保存禁止 | `password_hash` には必ず bcrypt ハッシュを格納する。平文を保存しない |
| bcrypt コストファクター | 12 以上を推奨（T000 §5-1 参照） |
| メール変更時の重複チェック | DB の `uk_email` ユニーク制約とアプリ層の重複チェックで二重防衛する |
| 旧パスワードの確認 | パスワード変更時は必ず現在のパスワードを照合し、セッション乗っ取りによる不正変更を防ぐ |
| SQL インジェクション | すべての操作はパラメータライズドクエリ（バインド変数）で実施する |

### 5-5. エラー応答定義

| ステータスコード | 内容 | 表示メッセージ |
|---|---|---|
| `400` | メール形式不正 / パスワード複雑性不足 | 各バリデーションメッセージ |
| `401` | 未認証 / 現在のパスワード誤り | `現在のパスワードが正しくありません` |
| `409` | メールアドレス重複 | `このメールアドレスはすでに使用されています` |
| `500` | サーバーエラー | `サーバーエラーが発生しました。しばらくしてからお試しください` |
