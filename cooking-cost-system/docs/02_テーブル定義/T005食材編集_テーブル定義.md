## 2. テーブル一覧

食材編集画面は **新規テーブルを作成しない**。  
既存の `items` テーブル（`item_type = 1`）に対して SELECT・UPDATE 処理を行う。

| テーブル名 | 論理名 | 操作種別 | 概要 |
|---|---|:---:|---|
| `items` | 統合商品マスタ | 参照 / 更新 | `id` で特定した食材レコードを取得し、編集内容で上書き更新する |

---

## 3. テーブル詳細（更新対象カラム）

### 3-1. `items` テーブル — 食材編集で操作するカラム

> テーブルの全体定義は `T003 食材追加_テーブル定義.md` を参照。  
> 以下は食材編集処理で特に関係するカラムを抜粋して記載する。

| カラム名 | 論理名 | データ型 | 編集可否 | 備考 |
|---|---|---|:---:|---|
| `id` | 商品ID | `INT` | 変更不可 | 編集対象の特定キー（`WHERE id = ?`） |
| `name` | 商品名 | `VARCHAR(255)` | 可 | 食材名の変更 |
| `store` | 購入先 | `VARCHAR(100)` | 可 | 購入先の変更。UI上で変更前/変更後を並べて表示 |
| `price` | 価格 | `DECIMAL(10,2)` | 可 | 価格の変更。UI上で変更前/変更後を並べて表示 |
| `quantity` | 内容量 | `DECIMAL(10,2)` | 可 | 内容量の変更 |
| `unit` | 単位 | `VARCHAR(20)` | 可 | 単位の変更 |
| `unit_price` | 単価 | `DECIMAL(10,4)` | **自動再計算** | `price` または `quantity` が変更された場合に API 層で `price ÷ quantity` を再計算して上書き |
| `genre` | ジャンル | `VARCHAR(50)` | 可 | ジャンルの変更（任意） |
| `description` | 説明/メモ | `TEXT` | 可 | メモの変更（任意） |
| `item_type` | 属性タイプ | `TINYINT` | 変更不可 | 食材編集では `item_type = 1` に固定。変更は許可しない |
| `updated_at` | 更新日時 | `TIMESTAMP` | 自動更新 | `ON UPDATE CURRENT_TIMESTAMP` により自動設定 |

#### 食材編集クエリの概要

```sql
-- ① 編集対象の食材を取得
SELECT id, name, store, price, quantity, unit, unit_price, genre, description
FROM   items
WHERE  id = :id
  AND  item_type = 1;   -- 食材のみ。誤って仕込み品・お品を更新しないための保護

-- ② 編集内容を上書き（unit_price はアプリ層で再計算してから渡す）
UPDATE items
SET    name        = :name,
       store       = :store,
       price       = :price,
       quantity    = :quantity,
       unit        = :unit,
       unit_price  = :unit_price,   -- price ÷ quantity の再計算値
       genre       = :genre,
       description = :description
WHERE  id = :id
  AND  item_type = 1;
```

## 5. 補足事項

### 5-1. unit_price の再計算ルール

`price` または `quantity` のいずれかが変更された場合、必ず以下の式で `unit_price` を再計算してから UPDATE すること。

```
unit_price = price ÷ quantity  （小数点第4位まで保持）
```

再計算をアプリ層で行う理由：現フェーズでは GENERATED COLUMN を使用していないため。  
将来的には DB 側の GENERATED COLUMN に移行してデータ整合性を DB レベルで保証する予定。

### 5-2. 関連データへの影響

食材の価格・量・単価を変更した場合、それを材料として使用している仕込み品・お品の原価計算に影響する。

| 関連データ | 対応方針 |
|---|---|
| `item_relations.cost`（仕込み・お品の構成コスト） | 食材編集時に関連する `item_relations` の `cost` を再計算して UPDATE する（アプリ層で実施） |
| 親の `items.price`（仕込み品・お品の総コスト） | 構成コストの合計を再集計して親 item の `price` を更新する |

### 5-3. セキュリティ

| 観点 | 方針 |
|---|---|
| 権限チェック | API 層で JWT トークンを検証し、認証済みユーザーのみ UPDATE を許可する |
| `item_type` の固定 | UPDATE 文に `AND item_type = 1` 条件を必ず付与し、異なる種別への誤更新を防ぐ |
| SQL インジェクション | すべての値はパラメータライズドクエリ（バインド変数）で渡す |
| 入力バリデーション | `price >= 0`・`quantity > 0` を API 層で検証（DB の CHECK 制約と二重防衛） |
