# レシピ構成 (item_relations) テーブル定義
（旧：仕込み テーブル定義）

## 1. 概要
「仕込み品（`item_type=2`）」を「食材（`item_type=1`）」から作成したり、「お品（`item_type=3`）」を「仕込み品」から作成したりする際の、「何を使って何を作ったか」の構造（BOM：部品表）を保存するための `item_relations` テーブルを使用する。
従来の `dish_ingredients` などを発展・統合した中間テーブルである。

## 2. テーブル一覧
| 論理名 | 物理名 | 役割・概要 |
| :--- | :--- | :--- |
| **レシピ構成関連** | `item_relations` | その親アイテムを作るために、「どの子アイテム(`child_item_id`)」を「どれだけの量(`amount`)」消費したかを結びつける。 |

## 3. テーブル詳細

### 3.1 レシピ構成関連 (`item_relations`)
| カラム名 (物理名) | データ型 | NULL | 備考 |
| :--- | :--- | :--- | :--- |
| `id` | `INT` | ✕ | AUTO_INCREMENT (主キー) |
| `parent_item_id` | `INT` | ✕ | 外部キー：紐づく親アイテムID (`items.id`) |
| `child_item_id`| `INT` | ✕ | 外部キー：紐づく子となる使用アイテムのID (`items.id`) |
| `amount`| `DECIMAL(10,2)`| ✕ | この親レシピにおける子アイテムの使用量 |
| `cost` | `DECIMAL(10,2)`| ✕ | 選択子アイテムの単価 × `amount` の算出結果 |
| `created_at` | `TIMESTAMP` | ✕ | レコード作成時刻 |

* **制約・インデックス**:
  - `fk_relation_parent` (`parent_item_id` -> `items(id)`)
  - `fk_relation_child` (`child_item_id` -> `items(id)`)
  - 重複登録防止の複合ユニークキー `uk_parent_child` (`parent_item_id`, `child_item_id`)

## 4. DDL（CREATE TABLE文）
```sql
-- レシピ構成テーブル
CREATE TABLE `item_relations` (
    `id` INT NOT NULL AUTO_INCREMENT,
    `parent_item_id` INT NOT NULL COMMENT '親アイテムID（作られる側）',
    `child_item_id` INT NOT NULL COMMENT '子アイテムID（使われる側）',
    `amount` DECIMAL(10,2) NOT NULL COMMENT '使用量',
    `cost` DECIMAL(10,2) NOT NULL COMMENT '使用コスト',
    `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '作成日時',
    PRIMARY KEY (`id`),
    CONSTRAINT `fk_relation_parent` 
        FOREIGN KEY (`parent_item_id`) REFERENCES `items` (`id`) 
        ON DELETE CASCADE ON UPDATE CASCADE,
    CONSTRAINT `fk_relation_child` 
        FOREIGN KEY (`child_item_id`) REFERENCES `items` (`id`) 
        ON DELETE RESTRICT ON UPDATE CASCADE,
    INDEX `idx_parent` (`parent_item_id`),
    INDEX `idx_child` (`child_item_id`),
    UNIQUE KEY `uk_parent_child` (`parent_item_id`, `child_item_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='レシピ構成・親子関係';
```

## 5. 補足事項・特記事項
* **合計コストの保持と集計**:
  フロントエンド画面上で計算される「子アイテムが掛かった全コストの合計」は、保存時に親の `items.price` に書き込まれることで計算キャッシュの役割を果たす。この結果を子から取得し直すためのテーブルが本件の `item_relations` である。
* **無限階層の構築可能**:
  親も子もすべて `items` マスタから引いてくるため、「食材」→「仕込みソース」→「別な仕込み調理済肉」→「最終的なお品」のように、複雑に組み上がるレシピに対しても、プログラム構造を変えることなく再帰的に部品表（BOM）を参照可能となる。
