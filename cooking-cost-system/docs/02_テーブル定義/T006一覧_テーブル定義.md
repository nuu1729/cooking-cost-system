## 2. テーブル一覧

一覧画面は **新規テーブルを作成しない**。  
統合商品マスタ（`items`）を `item_type` でタブ別に絞り込んで参照し、削除処理時に整合性を扱う。

| テーブル名 | 論理名 | 操作種別 | 概要 |
|---|---|:---:|---|
| `items` | 統合商品マスタ | 参照 / 削除 | タブに応じて `item_type` でフィルタし、一覧表示・削除を行う |
| `item_relations` | レシピ構成 | 参照 | 削除前の依存チェックに使用する |

---

## 3. テーブル詳細（参照）

### 3-1. `items` テーブル — タブ別フィルタ

> テーブルの全体定義は `T003 食材追加_テーブル定義.md` を参照。  
> 一覧画面ではタブに応じて `item_type` の値を変えて同一テーブルを参照する。

| タブ | `item_type` 値 | 旧テーブル名（廃止） | 表示する主なカラム |
|---|:---:|---|---|
| 食材タブ | `1` | `ingredients`（廃止） | `id`, `name`, `unit_price`, `store` |
| 仕込みタブ | `2` | `dishes`（廃止） | `id`, `name`, `price`（総コスト）, `description`, `genre` |
| お品タブ | `3` | `completed_foods`（廃止） | `id`, `name`, `price`（総コスト） |

> 旧テーブル名（`ingredients` / `dishes` / `completed_foods`）は統合BOM設計への移行に伴い廃止済み。  
> 以降はすべて `items` テーブルの `item_type` カラムで種別を区別する。

#### 一覧取得クエリの概要

```sql
-- タブに応じて :item_type を 1 / 2 / 3 で切り替える
SELECT id, name, item_type, store, price, unit_price, genre, description, updated_at
FROM   items
WHERE  item_type = :item_type
ORDER BY name ASC;
```

### 3-2. `item_relations` テーブル — 削除前の依存チェック

> テーブルの全体定義は `T007 仕込み_テーブル定義.md` を参照。  
> 旧テーブル名（`dish_ingredients` / `food_dishes`）は廃止済み。

削除しようとした `items` レコードが他の品目の材料として使われていないかを確認するために参照する。

```sql
-- 削除対象の item_id が子として使われているか確認
SELECT COUNT(*) AS ref_count
FROM   item_relations
WHERE  child_item_id = :item_id;
```

## 5. 削除処理の整合性制御

一覧画面の削除アイコン押下時は、RDBMS の外部キー制約に加えてアプリケーション層でも事前チェックを行う。

### 5-1. 食材（`item_type = 1`）の削除ルール

| チェック内容 | 判定 | エラーメッセージ例 |
|---|---|---|
| `item_relations.child_item_id` に該当する行が存在する | 削除不可 | `「この食材は『合わせ味噌』で使用中のため削除できません」` |
| 依存行なし | 削除許可 | — |

```sql
-- アプリ層での事前チェック
SELECT ir.id, i.name AS parent_name
FROM   item_relations ir
JOIN   items i ON i.id = ir.parent_item_id
WHERE  ir.child_item_id = :item_id
LIMIT  1;
-- 1件以上ヒット → 削除拒否（エラーメッセージに親の name を使用）
```

### 5-2. 仕込み品（`item_type = 2`）の削除ルール

| チェック内容 | 判定 | エラーメッセージ例 |
|---|---|---|
| `item_relations.child_item_id` に該当する行が存在する | 削除不可 | `「この仕込み品は『唐揚げ定食』で使用中のため削除できません」` |
| 依存行なし | 削除許可 | — |

> 仕込み品を削除するとその `item_relations` 行（子構成）は `ON DELETE CASCADE` により自動削除されるが、  
> 他の親の `child_item_id` として使われている場合は `ON DELETE RESTRICT` により DB レベルでも拒否される。  
> アプリ層で先にチェックして、ユーザーにわかりやすいメッセージを返すこと。

### 5-3. お品（`item_type = 3`）の削除ルール

| チェック内容 | 判定 | 備考 |
|---|---|---|
| 現フェーズでお品は他の品目の材料にならない | 削除許可 | 将来階層レシピに対応する場合は仕込み品と同様のチェックが必要 |

### 5-4. 削除実行クエリ

```sql
-- 事前チェックを通過した場合のみ実行
DELETE FROM items WHERE id = :item_id;
-- item_relations の子行は ON DELETE CASCADE で自動削除される
```

## 6. セキュリティ

| 観点 | 方針 |
|---|---|
| 権限チェック | API 層で JWT を検証し、認証済みユーザーのみ DELETE を許可する |
| 削除の二重確認 | フロントエンドで削除確認モーダルを表示し、誤操作を防止する |
| SQL インジェクション | `WHERE id = :item_id` および `WHERE item_type = :item_type` はパラメータライズドクエリで実施 |
| `item_type` の検証 | API 層で受け取った `item_type` 値が 1/2/3 のいずれかであることを検証する |
