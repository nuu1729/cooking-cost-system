## 2. テーブル一覧

ログイン機能では **新規テーブルは作成しない**。  
`users` テーブル（T000で定義済み）を参照・更新して認証処理を行う。

| テーブル名 | 論理名 | 操作種別 | 概要 |
|---|---|:---:|---|
| `users` | ユーザーマスタ | 参照 / 更新 | メールアドレス・パスワードハッシュで認証し、`last_login` を更新する |

---

## 3. テーブル定義（参照）

### 3-1. `users` テーブル（ユーザーマスタ）— ログイン時の使用カラム

> テーブル定義の全体は `T000 アカウント作成_テーブル定義.md` を参照。  
> 以下はログイン処理で特に関係するカラムを抜粋して記載する。

#### ログイン認証で使用するカラム

| カラム名 | 論理名 | データ型 | ログイン処理での役割 |
|---|---|---|---|
| `id` | ユーザーID | `INT` | JWTペイロードへ埋め込む識別子 |
| `email` | メールアドレス | `VARCHAR(255)` | ログイン識別子。入力値と照合する（`WHERE email = ?`） |
| `password_hash` | パスワードハッシュ | `VARCHAR(255)` | bcryptで入力パスワードと照合する |
| `role` | ロール | `ENUM('user','admin')` | JWTペイロードへ埋め込む権限情報 |
| `is_active` | 有効フラグ | `TINYINT(1)` | `0`（無効）の場合はログインを拒否する |
| `last_login` | 最終ログイン日時 | `DATETIME` | ログイン成功時に現在日時で **UPDATE** する |

#### ログイン認証クエリの概要

```sql
-- ① メールアドレスでユーザー検索
SELECT id, email, password_hash, role, is_active
FROM   users
WHERE  email = :email
LIMIT  1;

-- ② パスワード照合はアプリ層（bcrypt.verify）で実施
-- ③ 認証成功後、最終ログイン日時を更新
UPDATE users
SET    last_login = NOW()
WHERE  id = :id;
```

## 5. 補足事項

### 5-1. 認証フロー（ログイン成功時）

```
① POST /api/auth/login  { email, password }
       │
       ├─ users テーブル検索（WHERE email = ?）
       │       └─ 未ヒット → 401（ユーザーが存在しない）
       │
       ├─ is_active = 0 → 401（アカウント無効）
       │
       ├─ bcrypt.verify(password, password_hash)
       │       └─ 不一致 → 401（パスワード誤り）
       │
       ├─ users.last_login を NOW() に UPDATE
       │
       └─ JWT 発行 → レスポンスに { token } を返却
```

### 5-2. JWT ペイロード仕様

| キー | 値 | 出所カラム |
|---|---|---|
| `sub` | ユーザーID | `users.id` |
| `email` | メールアドレス | `users.email` |
| `role` | ロール | `users.role` |
| `iat` | 発行日時（UNIX時刻） | サーバー生成 |
| `exp` | 有効期限（UNIX時刻） | サーバー生成（例: 発行から24時間） |

> トークン自体はDBに保存しない（ステートレス運用）。  
> フロントエンドは受け取ったトークンを `localStorage` の `authToken` キーへ保存する。

### 5-3. エラー応答定義

| ステータスコード | 内容 | 表示メッセージ |
|---|---|---|
| `400` | バリデーションエラー（形式不正） | `有効なメールアドレスを入力してください` |
| `401` | 認証失敗（ユーザー不在・パスワード不一致・アカウント無効） | `メールアドレスまたはパスワードが正しくありません` |
| `500` | サーバー内部エラー | `サーバーエラーが発生しました。しばらくしてからお試しください` |

> セキュリティ上、「ユーザーが存在しない」「パスワード誤り」「アカウント無効」は **同一の 401 メッセージに統一** し、情報を外部に漏らさない。

### 5-4. 将来の拡張予定

| 項目 | 対応案 |
|---|---|
| ログイン試行ロック | 一定回数失敗したら一時ロック。`login_failed_count INT` / `locked_until DATETIME` カラムを `users` テーブルへ追加予定 |
| リフレッシュトークン | `refresh_tokens` テーブル（`id`, `user_id`, `token_hash`, `expires_at`, `created_at`）を追加し、長期ログイン維持に対応予定 |
| OAuth連携ログイン | `oauth_providers` テーブルを追加し、`users.id` に外部キーで紐付ける予定（Google / Apple） |
| 2段階認証 | ログイン後に TOTP コードを検証するフローを追加予定 |
