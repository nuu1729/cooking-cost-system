## 1. プロジェクト概要

本プロジェクトは、飲食店の料理原価を正確かつ迅速に計算するためのWEBシステムを構築するプロジェクトです。
AI（Antigravity）を最大限に活用し、高速かつ高品質な開発を行うことを目的としています。

### 1.1 本手順書の目的

本手順書は、開発担当者が「AIを使ってどのように開発を進めるか」を具体的に示したガイドラインです。
単なるツールの使い方だけでなく、AIへの指示の出し方（プロンプト）、AIが生成した成果物のレビュー方法、品質を担保するためのチェックポイントを網羅しています。

### 1.2 ターゲット読者

- 本プロジェクトに参画する全ての開発者（フロントエンド・バックエンド）
- AIを使った開発が初めて、または慣れていない方

### 1.3 システムアーキテクチャ

本開発のシステムアーキテクチャは以下のとおりです。

| レイヤー | 技術スタック | 選定理由・備考 |
| :--- | :--- | :--- |
| **Frontend** | **Vite + React** | 高速なビルドツールと開発サーバー。HMRによる快適な開発体験を提供。 |
| **Language** | **TypeScript** | 型安全性によるバグ低減。バックエンド（OAS）との型同期を容易にするため厳格な型チェックを行う。 |
| **Styling** | **Tailwind CSS** | ユーティリティファーストCSS。**AIが最も正確にコード生成できる**スタイリング手法であるため採用。 |
| **State Management** | **Jotai** | Atomicな状態管理。シンプルで学習コストが低く、TypeScriptとの親和性が高い。 |
| **Routing** | **React Router** | SPAのルーティング。宣言的なルーティング定義により保守性が高い。 |
| **Backend** | **Python + Flask** | シンプルかつ軽量なフレームワーク。AI（LLM）との親和性が高く、将来的なAI機能拡張も見据えて採用。 |
| **Database** | **MySQL** | 信頼性の高いリレーショナルデータベース。 |

## 2. 開発担当者の役割

### 2.1 要件定義・設計担当者

- **ユーザー要件のヒアリング**: 「何を実現したいか」を明確化。
- **要件定義書の作成**: 開発チームとユーザーの合意形成。
- **画面設計**: Figma等を用いたUI設計。
- **API仕様・テーブル設計**: OpenAPI, DDLの策定。
- **AIとの協業**: 要件整理や設計レビューにAIを活用。

### 2.2 フロントエンド担当者

- **UIの実装**: デザインの再現と、UX（ローディング、エラー表示等）の作り込み。
- **API仕様へのフィードバック**: 「使いやすいAPI」をバックエンドに要求。
- **フロントエンドファースト**: バックエンドを待たずにMockを活用して開発を先行させる。

### 2.3 バックエンド担当者

- **APIの実装**: フロントエンドの要求に応えるAPIの提供。
- **ビジネスロジックの実装**: 堅牢なバリデーションと計算処理。
- **仕様の明確化**: AIが解釈可能な明確なコードとドキュメントを作成。

### 2.4 AIの役割と立ち位置

- **AI (Coder)**: 指示通りにコードを書く「優秀な新人」。指示が曖昧だと間違える。
- **人間 (Director/Reviewer)**: 的確な指示を出し、成果物を責任を持って検証する「監督」。
- **重要**: 作成とレビューは別のチャットセッションで行うことで、AIの自己肯定バイアスを防ぐ。

## 3. 開発環境のセットアップ

### 3.1 プロジェクト共通のセットアップ

#### Antigravityのセットアップ
1. 推奨拡張機能（Japanese Language Pack, ESLint, Prettier, Python等）をインストール。
2. `AGENTS.md` および `rules/` 配下のルールファイルを活用できる状態にする。

### 3.2 フロントエンド開発環境のセットアップ

1. **Node.js**: 指定バージョンを使用（`package.json`参照）。
2. **依存関係インストール**:
   ```bash
   cd frontend
   npm install
   ```
3. **環境変数設定**:
   `.env.example` をコピーして `.env.local` を作成し、必要な値を設定。
4. **起動確認**:
   ```bash
   npm run dev
   ```

### 3.3 バックエンド開発環境のセットアップ

1. **Python**: バージョン 3.12系 を使用。
2. **パッケージ管理**: `pip` または `uv` を使用。
3. **セットアップ**:
   ```bash
   cd backend
   # 仮想環境作成
   python -m venv .venv
   # 依存関係インストール
   pip install -r requirements.txt
   ```
4. **環境変数設定**:
   `.env.example` をコピーして `.env` を作成し、DB接続情報等を設定。
5. **起動確認**:
   ```bash
   python app.py
   ```

## 4. 開発ワークフロー全体像

### 4.1 基本サイクル

AI駆動開発では、以下のサイクルを高速に回します。

1. **Prompt (指示)**: コンテキスト（ファイル、ルール）を与えて具体的に指示。
2. **Generate (生成)**: AIによるコード・ドキュメント生成。
3. **Self-Check (自己検証)**: 人間による動作確認とコードチェック。
4. **AI Review (AIレビュー)**: 別セッションでAIに客観的レビューを依頼。
5. **Refine (修正)**: 指摘事項を修正。
6. **Approve (完了)**: マージ。

### 4.2 70点主義（イテレーティブ開発）

最初から完璧を目指さず、「まずは動くもの（70点）」をAIに作らせ、対話を通じて完成度を高めます。
これにより、プロンプト作成時間を短縮し、実際の成果物を見ながら具体的かつ建設的な改善が可能になります。

### 4.3 フロントエンドファースト（UI駆動開発）

データベース構造からではなく、「画面（UI）」を起点に開発します。
1. UIを作る（Mock活用）。
2. UIに必要なデータを特定する。
3. バックエンドAPIとDBを実装する。
これにより、手戻りを防ぎ、ユーザーにとって使いやすいシステムを実現します。

### 4.4 AI協調サイクル詳細

**重要: 作成とレビューのセッション分離**
- **作成セッション**: コードを書かせる。
- **レビューセッション**: 新しいチャットでコードを貼り付け、客観的にレビューさせる。

## 5. フェーズ別開発プロセス

本プロジェクトでは、各フェーズにおいてAIを最大限活用し、ドキュメントとコードの整合性を保ちながら進めます。

### 5.1 要件定義フェーズ

ユーザーの要望を明確化し、画面イメージとデータ構造を定義します。このフェーズでのAI活用が、後の手戻りを大きく減らします。

#### 5.1.1 業務要件と画面要件の確認
1. **ヒアリング**: ユーザー（またはプロダクトオーナー）から実現したい機能をヒアリングします。
2. **AI壁打ち（ユースケース抽出）**:
   - AIに対して「〇〇機能を作りたいが、どのようなユースケースが考えられるか？」「例外的な操作は何がありうるか？」と問いかけ、要件の抜け漏れを防ぎます。
   - 想定される「ユーザーの痛み（Pain Points）」をAIと洗い出します。

#### 5.1.2 画面設計（UI/UX）
1. **ワイヤーフレーム作成**: 必要に応じてFigmaや手書きで画面レイアウトを作成します。
2. **AIレビュー**:
   - 画面イメージをテキストまたは画像でAIに説明し、「使い勝手に問題はないか」「不足している項目はないか」をレビューさせます。
   - プロンプト例: 「この在庫入力画面で、ユーザーが入力を間違えやすいポイントはどこだと思う？」

#### 5.1.3 ドキュメント作成
以下のドキュメントを `docs/` 配下に整備します。AIに草案を作らせ、人間が修正するフローが効率的です。
- `01_画面設計書/`: 画面レイアウト、イベント定義、表示項目定義
- `02_テーブル定義/`: ER図、DDL
- `03_OpenAPI/`: APIエンドポイント定義

### 5.2 設計～単体テストフェーズ

フロントエンドとバックエンドは、API定義（OpenAPI）を契約（Contract）として並行して開発を進めます。

#### 5.2.1 フロントエンド開発フロー

**「動く画面」を最速で作る** ことが目標です。

1. **画面仕様書の詳細化**:
   - `docs/01_画面設計書` をもとに、バリデーションルールやエラーハンドリングを詳細化します。
   - AI活用: 「この画面項目のバリデーションルール（Zodスキーマ）を提案して」
   - AI活用: 「入力エラー時のメッセージ文言を考えて」

2. **コンポーネント実装 (UI First)**:
   - `src/components/ui` (汎用パーツ) → `src/components/features` (機能パーツ) の順で実装します。
   - Tailwind CSSを用いたスタイリングはAIに一任します。
   - プロンプト例: 「shadcn/uiのButtonコンポーネントを使って、青色の送信ボタンを作って。ホバー時は少し濃くして。」

3. **Mockデータによる動作確認**:
   - バックエンドが未完成でも、MSW (`src/mocks`) を使ってAPIレスポンスを模倣し、画面の動作を確認します。
   - **重要**: ここでUIの「使いにくさ」が見つかったら、直ちに修正します。バックエンド完成後では修正コストが高くなります。

4. **単体テスト (Vitest)**:
   - 複雑なロジック（HookやUtility）に対してテストを記述します。
   - AI活用: 「このカスタムフックのテストケースを網羅的に生成して」

#### 5.2.2 バックエンド開発フロー

**「堅牢で正確なAPI」** を作ることが目標です。

1. **API設計 (Schema First)**:
   - フロントエンドからの要求に基づき、OpenAPI (Swagger) でAPI仕様を確定させます。
   - リクエスト/レスポンスの型定義を明確にします。

2. **DDL作成・DB構築**:
   - `docs/02_テーブル定義` にDDLを作成し、マイグレーションを行います。
   - A5:SQL Mk-2等でER図を確認し、リレーションに矛盾がないかチェックします。

3. **実装 (Controller/Service/Model)**:
   - **Model**: SQLAlchemyモデルを定義します。
   - **Service**: ビジネスロジックを実装します。ここで業務ルールを担保します。
   - **Controller (Route)**: リクエストを受け取り、Serviceを呼び出し、統一されたフォーマットでレスポンスを返します。
   - AI活用: 「Userモデルを作成して。カラムはid, name, email...」
   - AI活用: 「Serviceクラスのスケルトンコードを作成して」

4. **単体テスト (pytest)**:
   - 正常系・異常系のテストケースを作成します。
   - カバレッジを意識し、AIに境界値や異常系のテストコードを書かせます。

---

### 5.3 結合・システムテストフェーズ

モジュールごとの動作確認が済んだ後、システム全体としての動作を検証します。

#### 5.3.1 結合テスト (Integration Test)
- **API連携**: フロントエンドとバックエンドを実際に接続し、データが正しく保存・取得できるか確認します。
- **シナリオテスト**: 「ログイン → 商品検索 → 在庫登録 → ログアウト」といった一連のユーザー操作が通るか確認します。
- **異常系確認**:
    - ネットワークエラー時の挙動
    - サーバーエラー時の画面表示
    - バリデーションエラー時のメッセージ

#### 5.3.2 品質管理（AIレビュー）
実装完了後、AIに以下の観点で最終レビューを依頼します。
- **セキュリティ**: SQLインジェクション、XSS、認証不備の可能性はないか。
- **パフォーマンス**: N+1問題や非効率なクエリはないか。
- **保守性**: 変数名は適切か、コードの重複はないか。

#### 5.3.3 完了基準
- すべてのテストケースがPassしていること。
- Lintエラー、型エラーがないこと。
- 既知のバグが解消されている、または回避策がドキュメント化されていること。
